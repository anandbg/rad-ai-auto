## Phase Details

### Phase 31: Provider Abstraction Layer
**Goal**: AI model selection is decoupled from route handlers, configurable via environment variables
**Depends on**: Nothing (first phase of v3.0; builds on existing Phase 30 infrastructure)
**Requirements**: INFRA-01, INFRA-02
**Success Criteria** (what must be TRUE):
  1. Changing AI_PROVIDER and AI_MODEL environment variables switches the model used for report generation without any code changes
  2. Route handlers call a generic model resolver instead of importing OpenAI directly -- swapping providers requires zero route file edits
  3. Application fails fast at startup with a clear error if required AI provider environment variables are missing
  4. All existing report generation and transcription functionality works identically (OpenAI still active, no behavior change)
**Plans:** 2/2 plans complete
Plans:
- [x] 31-01-PLAN.md — Provider registry and env-driven config module
- [x] 31-02-PLAN.md — Wire route handlers to use provider registry

### Phase 32: LLM Migration & Quality Validation
**Goal**: Radiology reports are generated by Groq Llama 4 Scout at ~96% lower cost with quality validated against GPT-4o baseline
**Depends on**: Phase 31
**Requirements**: GEN-01, GEN-02, GEN-03
**Success Criteria** (what must be TRUE):
  1. Report generation uses Groq Llama 4 Scout as the primary model, producing structured radiology reports (findings, impressions, recommendations) from the same inputs that previously went to GPT-4o
  2. System prompt is under 2K tokens with explicit reasoning chains and anti-hallucination rules adapted for open-source models
  3. A quality validation suite of 20+ test cases across MRI, CT, X-ray, and ultrasound modalities exists, comparing Llama 4 output against GPT-4o baseline reports
  4. SSE streaming continues to work for report generation with the new provider
**Plans:** 3/3 plans complete
Plans:
- [x] 32-01-PLAN.md — Register Groq provider in AI SDK registry
- [x] 32-02-PLAN.md — Create 25 test fixtures and quality validation functions
- [x] 32-03-PLAN.md — Adapt system prompt for Llama 4 Scout and switch defaults to Groq

### Phase 33: Transcription Migration
**Goal**: Voice transcription uses Groq Whisper v3 Turbo at ~89% lower cost with accurate medical terminology
**Depends on**: Phase 31
**Requirements**: TRANS-01
**Success Criteria** (what must be TRUE):
  1. Voice transcription uses Groq Whisper v3 Turbo instead of OpenAI Whisper, processing audio uploads and returning text
  2. Medical vocabulary hints are sent with transcription requests, improving accuracy for radiology terminology (e.g., "pneumothorax", "T2-weighted", "FLAIR sequence")
  3. Transcription latency and accuracy are comparable to or better than OpenAI Whisper for typical radiology dictation (30-120 second recordings)
**Plans:** 1/1 plans complete
Plans:
- [x] 33-01-PLAN.md — Switch transcription to Groq Whisper v3 Turbo with medical vocabulary hints

### Phase 34: Reliability & Cost Tracking
**Goal**: System automatically falls back to OpenAI when Groq fails, and cost tracking reflects actual provider-specific token usage
**Depends on**: Phase 32, Phase 33
**Requirements**: REL-01, COST-01
**Success Criteria** (what must be TRUE):
  1. When Groq returns an error or is unavailable, the system automatically routes to OpenAI GPT-4o for report generation and OpenAI Whisper for transcription -- with no user action required
  2. Cost tracking uses actual token counts from provider responses (not hardcoded estimates), with per-provider pricing rates
  3. Daily cost ceiling and per-user abuse detection work correctly with the new multi-provider cost structure (including higher costs during fallback)
**Plans:** 3/3 plans complete
Plans:
- [x] 34-01-PLAN.md — Provider-aware cost tracker (pricing module + usage-based trackCost)
- [x] 34-02-PLAN.md — Fallback chain utility + wire generate/templates routes with real token cost tracking
- [x] 34-03-PLAN.md — Transcription fallback + daily cost ceiling threshold update
